Illustrator

O layout deve ser gerado no Illustrator prioritariamente. Existe um arquivo modelo pronto para isso. Neste arquivo existem layers prontas com os elementos além das paletas igual à usada no sistema atual.

As cores da paleta são globais e alteram os elementos automaticamente. Para criar uma nova paleta siga o mesmo padrão, inclusive de nome, para que possa ser transcrito no arquivo de configuração do Front-Alego.

Quando alterar ou criar uma nova paleta, comece pela quarta cor (a cor do meio) de cada paleta e crie as três variantes mais claras (as 3 primeiras) e as três variantes mais escuras (as 3 últimas).

Regras de padronização

BEM - Bloco, Elemento, Modificador

Estamos utilizando a regra do BEM para nomear as classes.

Bloco
O bloco é o elemento pai, principal. Procure usar nome relevante e de fácil identificação. Se precisar separar palavras do nome, use apenas um traço: -
Elemento
É o elemento filho do bloco. Ele deve aparecer logo depois do bloco e separado com dois sublinhados: __
Modificador
Uma variação do bloco ou elemento. Deve ser separada com dois traços: --

Exemplo

Imagine a seguinte estrutura: você tem uma caixa com um cabeçalho, um corpo com lista e formulário e um rodapé com ações.

Os nomes seriam:

  • Caixa: .box
  • Cabeçalho: .box__cabecalho ou .box__header
  • Corpo: .box__main ou .box__corpo
  • Rodapé: .box__footer ou .box__rodape.
  • Formulário: .box__form ou .box__main__form. A primeira opção é melhor, a menos que você tenha mais de um formulário dentro da caixa e ela tenha formatações diferentes.
  • Tabela: .box__tab ou .box__main__tab.

Se tivéssemos uma caixa com variação (exemplo: urgente), ela seria .box--urgente.

Se você tivesse uma tabela com os dados maior poderia ser .box__tab--maior.

Filhos pegam o nome variante do pai?

Não. Os elementos filhos fazem uso do nome do elemento pai sem o modificador e o CSS é formatado com uso de descendente:

HTML

<div class="box--urgente">

<section class="box__tab">

<...>

</section>

</div>
CSS

.box__tab { ... } /* TODOS os .box__tab */

.box--urgente { ... }

.box--urgente .box__tab { ... } /* Os box__tab que são filhos do .box--urgente

É melhor desta forma do que termos uma classe .box--urgente__tab. Primeiro porque nem sempre teremos configuração específica do elemento filho quando ele estiver dentro do pai (e teremos que extender uma classe com a outra). Segundo porque diminui a quantidade de classes usadas no sistema.

O grande defeito dessa abordagem é o uso de seletores descendentes (ver mais abaixo).

BEM no SASS

O SASS facilita a criação dos elementos filhos e variantes, inclusive com a opção de retrair e expandir o código diretamente no editor. Para isso funcionar basta usar o símbolo de ampersand (&).

SASS

.box {

&__header { ... }

&__main { ... }

&--urgente {

@extend .box;

...

}

}

Isso vai resultar em:

CSS

.box, .box--urgente { ... }

.box__header { ... }

.box__main { ... }

.box--urgente { ... }

Nomes de classes

Jamais use nas classes específicas elementos de formatação. Estes devem aparecer como classe adicional ou modificador de classe (--mod).

Exemplo: se você tiver um card que deve aparecer alinhado na esquerda da página, não use .card-alinha-esquerda e sim .card--esquerda ou ainda duas classes (.card e .float-left).

Para elementos comuns de formatação use as seguintes simplificações:

bt
botão
tab
tabela
ico
ícone, símbolo
form
formulário

Organização dos arquivos SCSS

Como explicado aqui os arquivos são armazenados em pastas específicas. Elementos gerais que fujam do padrão do Front-Alego devem ser salvos em arquivos separados dentro da pasta /extra. Um exemplo disso é tabelas. Mixins e funções também vão nesta pasta. Se você tiver classes genéricas (que são usadas em todo o seu sistema e não em parte específicas) é recomendado que você crie um arquivo chamado classes nessa pasta.

Comentários

O arquivo deve ser totalmente comentado ao estilo SASS para identificar seus trechos.

Ele deve abrir com um comentário maior com o nome do sistema, o usuário que criou o arquivo e a data de criação. Veja como exemplo o CSS usado neste arquivo:

================================================================================

FRONT-ALEGO

--------------------------------------------------------------------------------

Arquivo de CSS modelo utilizado na documentação. Use o arquivo SCSS base desta

pasta (info.scss) para ver como foi utilizado e também para acrescentar

eventuais elementos para visualizar o arquivo final.

Não se esqueça de compilar

--------------------------------------------------------------------------------

Autor: Alessandro Guarita

Criado em: 23/11/2015

================================================================================

Cada parte do arquivo deve ser separada com comentários usando sinal de igual até aproximadamente 80 caracteres. E o nome deve ser em caixa alta. Exemplo:

// =============================================================================

// HEADER

// =============================================================================

Sub-seções são separadas da mesma forma acima, mas com uso de traço no lugar do igual. No comentário coloque o nome do elemento pai e separe com pipe (|):

// -----------------------------------------------------------------------------

// HEADER | MENU

// -----------------------------------------------------------------------------

Ordem dos elementos

Os elementos dentro de um arquivo devem obedecer a seguinte ordem (sempre comentado, conforme explicado acima):

  1. Variáveis e mapas gerais
  2. Classes gerais (que só usam neste arquivo e não são de nenhuma seção específica. Se for usada em mais de um arquivo, salve em um arquivo em extra/classes.scss)
  3. Seções do site (header, menu, principal, footer)
Variáveis e placeholders podem aparecer dentro de uma seção do site no CSS, caso não se repitam em nenhum lugar. Neste caso ficaria assim para facilitar a localização e modificação.

Atualização emergencial

Você conhece a situação: você tem que fazer uma alteração pra ontem. Por mais que o objetivo do Front-Alego™ seja organizar os arquivos, isso pode levar um tempo que não se dispõe. Neste caso haverá uma quebra das regras, temporariamente. Mas mesmo quebrando regras, temos regras, porque aqui não é a Casa da Mãe Joana. Siga os passos:

  1. Crie um arquivo chamado ajustes.scss na pasta extras
  2. Escreva um comentário indicando para quê é essa alteração e seu nome
  3. Escreva as regras. Não tenha medo de usar !important e outras regras escrotas pouco apropriadas
  4. No final do arquivo principal do seu CSS que a regra tem que ser aplicada dê um import para o arquivo:

    @import "extras/ajustes"
  5. Teste
  6. Commite o trabalho
  7. Depois que a corda saiu do seu pescoço insira as alterações nos lugares corretos e com as regras de boas condutas aplicadas
  8. Delete o @import para o arquivo de ajustes (pode deletar o conteúdo dos ajustes também)

Seletores descendentes e aninhamento de código

Evite o máximo possível CSS com seletor descendente (.classe1 .classe2). Este seletor diminui a performance do browser e aumenta a especifidade do código. Por esta mesma razão evite grandes aninhamentos de código no SASS. Esta regra, no entanto, tem exceções (uma delas já mostrada acima, no BEM). Mas mesmo assim, tenha no máximo dois descendentes e aninhamento (exemplo: .tab td e não .tab tr td).

Medidas relativas

Para que o sistema seja adaptável, sempre use medidas relativas. Jamais use valores em pixels (a exceção é tamanho de imagens e vídeos externos). Para os demais casos use em, rem, %, vw, etc.

REM vs. EM

Sabendo da diferença entre ambos (rem é relativo ao tamanho da fonte do root e em é relativo ao tamanho da fonte do elemento pai), use os dois quando adequado. Dicas:

  • Tamanho da fonte que é para ser relativa ao tamanho pai, use em. Exemplo: um link, um destaque ou mesmo todo um bloco de textos.
  • Tamanho de fonte que é para se manter com o mesmo tamanho, independente do tamanho do elemento pai, use rem. Exemplo: título da página.
  • Margens costumam ser mais adequado utilizar rem para manter a consistência.
  • Paddings, por outro lado, é mais adequado usar em para que o espaço interno cresça conforme se aumenta a fonte. Exceção: se você tem um elemento com dois tamanho de fonte diferentes, manter o padding em rem ajuda a manter a consistência e alinhamento (exemplo: tabela).
Existem funções no bourbon que convertem o valor de pixels para rem ou em. Basta usar a função rem() ou em()

Media Queries

O uso de Media queries para fazer sites responsivos não usa mais medidas em pixels e sim em rem, pois assim o layout se adapta inclusive ao zoom dado pelo usuário.

Ver mais detalhes sobre isso neste artigo.

A ideia básica é parar de usar medidas de dispositivos específicos (que a cada momento tem novos) e trabalhar com a medida da tela em si.

Sempre use o sistema de media-query descrito aqui (fazendo uso da função breakpoint). E nunca, jamais, escreva uma medida que não esteja descrita no mapa $breakpoints.

Ordem das propriedades dentro de um seletor

Calma, não precisa organizar TODOS as propriedades com uma regra específica. Mas as propriedades devem seguir a seguinte regra básica:

  1. Primeiro aparecem os @extend (seja para um placeholder ou para outra classe). Isso é importante pois você pode incluir uma propriedade que altera o que você extendeu.
  2. Depois aparecem os @include para os mixins
  3. Se você duplicar regras para teste (exemplo: dois background diferentes) lembre-se de comentar a que não está sendo usada ou, melhor ainda, deletar.

Uso de placeholder (%placeholder) e @extend

O uso de placeholders deve ser diminuído ao mínimo necessário. Apesar de eles deixarem o arquivo menor ele exige um pouco mais de performance (pois faz uso do seletor ,). Além do mais usamos o gzip que compacta melhor blocos repetidos de texto.

Outro grande defeito do uso de placeholders é que o resultado final do CSS não fica de leitura fácil nem mesmo no inspetor (as classes ficam separadas demais).

Nossa, isso quer dizer que eu não devo usar nunca mais um placeholder? Calma, jovem padawan, claro que você vai usar. Mas comedidamente e dentro do contexto. Imagine que você está criando um padrão para dentro de uma coluna lateral que incluir uma cor de fonte, uma fonte diferente e tamanho de fonte diferente. Três elementos vão usar desta configuração (parágrafo, tabela e lista). Neste caso você cria um placeholder dentro do contexto (logo no início do bloco). E comente indicando um placeholder. Se você fuçar no código vai ver que o Front-Alego faz uso de placeholders e @extends pontuais (botões e medidas de colunas de tabela são exemplos).

Extender classes também deve ser evitado. Mas isso, obviamente, tem uma exceção clara: com o uso do BEM você acaba extendo classes (inclusive no próprio exemplo do BEM mostra isso). Nos demais casos, evite.

HTML básico

Sempre faremos uso dos novos tags estabelecidos pelo HTML 5. O código de uma página básica seria aproximadamente o seguinte:

HTML

<body>

<header>...</header>

<nav>...</nav>

<main>...</main>

<aside>...</aside>

<footer>...</footer>

</body>

Como pode ser visto, cada um dos tags HTML aparecem como filhos diretos do <body>. Isso facilita a formatação do layout no CSS e também melhora a aplicação de media query para fazer o site responsivo.

header
O cabeçalho do site/sistema. Normalmente contém o nome e logo do sistema e botão de entrar/sair do sistema
nav
O menu do site/sistema. Este tag deve deve ser filho direto do <body> para ter mais flexibilidade na formatação final.
main
A área principal do site/sistema. Note que a regra do HTML 5 diz que só pode haver um <main> dentro do código.
aside
Área complementar do site/sistema. Pode ser uma área de banner, informações extras e assim por diante. tag opcional
footer
O rodapé do site/sistema. Normalmente um atalho para as principais seções do mesmo. tag opcional

Cores

O Front-Alego™ trabalha com as cores de duas formas: através de uma paleta de cores que pode ser chamadas por nomes específicos (exemplo básica) e através de um tema que pode alterar todas as cores do sistema, sem precisar mudar a paleta, mantendo a harmonia.

As cores ganham variações clara e escura através de uma função baseada no Bourbon.

Todas as cores devem ser definidas aqui. Jamais chame uma cor em outro CSS usando hexadecimal, nome de cor ou variável!

Cada cor representa uma matiz principal e deve ter um nome fácil de lembrar, com um comentário indicando qual a cor (apenas para facilitar a lembrança, é possível ver todas as paletas aqui). Sempre comece alterando as já estabelecidas e acrescente novas conforme a necessidade.

Temas

Neste arquivo ficam os temas de cores do sistema. Você pode criar vários outros temas e alterar no arquivo de configuração. Ao escolher um tema, as cores do sistema mudam de acordo.

Assim como acontece com a $paleta, os temas estão agrupados dentro de um mapa chamado $temas. Dentro deste mapa teremos um sub-mapa para cada tema, que são (por padrão):

principal
O tema padrão do sistema. Se você não for usar outro tema, basta alterar somente este.
cinza
Este tema faz uso apenas de tons de cinza. Pode ser útil em um momento de prototipação.
alto-contraste
Um tema com cores bem contrastantes, pensando em acessibilidade.
bandeira
Faz uso das cores da bandeira de Goiás. Para utilizar em sistemas mais institucionais ou eventos do Estado.
rosa
Tema com predominância das cores rosa. Para o Outubro Rosa, por exemplo.
azul
Tema com predominância das cores azuis. Para o Novembro Azul, por exemplo.

Acrescente novo mapa conforme a necessidade. Duplique o mapa principal e trabalhe nele (um exemplo de possibilidade para isso é um tema de Natal).

Dentro de cada tema existem diversos mapas com suas cores específicas. Podemos dividir estes mapas em três categorias, elementos genéricos (que são usados em qualquer contexto), elementos específicos (que são usados dentro de um contexto específico e já está sendo sendo usado dentro das configurações do Front-Alego, em arquivos próprios. Ex.: tabela) e elementos do sistema que são os elementos criados para o seu sistema (exemplo: menu, cabeçalho). Cada uma destas categorias estão separadas por um comentário com seu nome.

z-index

config/z-index.scss

Nunca use z-index diretamente no CSS, pois isso deixa a manutenção ruim, além de resultar em uso de números aleatórios (como o clássico z-index: 99999;).

No arquivo config/z-index.scss você vai inserir os elementos com os nomes dele. Para isso inclua os elementos no mapa $z. A ordem que os elementos aparecem é o valor do z-index. O primeiro elemento vai receber z-index: 1 e assim por diante. O mapa já vem com alguns elementos pensando nessa ordem:

SASS

$z: (

conteudo,

rodape,

topo,

modal,

toaster,

);

No entanto, colocar estes elementos não basta, você deve indicar no seletor qual o nível que ele ocupa. Para isso, chame através da função z(). Dentro do parênteses você coloca o mesmo nome do elemento do mapa $z. Por exemplo:

SASS

.header {

z(topo);

...

}

Se você olhar no mapa $z, vai perceber que topo é o terceiro elemento, o que resulta no seguinte CSS:

CSS

.header {

z-index: 3;

...

}

Sendo responsivo

Sempre que for chamar uma media query, você não o fará da maneira tradicional. Para chamar uma media query você vai chamar através da função breakpoint()

breakpoint()
Procura o valor de medida no mapa $breakpoints para retornar um valor para a media query.

O mapa $breakpoints contém os valores a serem usados dentro da media-query, que são:

pequeno
Atinge dispositivos de tela pequena, como smartphones.
medio
Atinge a medida média, que pode ser tablet em pé (retrato) ou smartphone na horizontal.
grande
Pega de telas de tablet na horizontal para cima.
extra-grande
Pega medidas de notebook para cima.

Quando você especifica min-width você vai estar selecionando um valor maior que aquele. Portanto, min-width: breakpoint(pequeno) vai pegar acima da tela pequena (mas ela não está incluída), assim como min-width: breakpoint(medio) vai pegar telas maiores que as médias.

Quando você especifica um valor com a de max-width ele vai pegar daquela medida para baixo. Por exemplo max-width: breakpoint(pequeno) vai pegar somente telas pequenas, enquanto max-width: breakpoint(medio) vai pegar de tamanho médio para baixo.

Para chamar a media query você pode digitar @media da maneira tradicional ou utilizar o mixin media-tipo:

media-tipo(breakpoint, medida)
Este mixin chama a media query pensando na técnica mobile first (utilizando min-width) como Valor padrão. O breakpoint é um dos descritos na lista acima.

Como trabalhar Media Query

No caso de mobile first basta usar o mixin omitindo a medida:

SASS

.classe {

...

...

}

}

Se preferir não trabalhar com o mixin, basta escrever assim:

SASS

.classe {

...

...

}

}

Se não for o caso de mobile first, pode ser:

SASS

.classe {

...

...

}

}

Ou assim, se for sem o mixin:

SASS

.classe {

...

...

}

}

Grupos de Formulários

Os formulários são agrupados em um sistema similar ao de grids. Primeiro você determina se vai usar este sistema ou se vai determinar o tamanho dos formulários na unha (essa alteração é feita no arquivo de configuração).

Para determinar o tamanho usamos flex-box, portanto é necessário perceber que teremos dois níveis de configuração: o elemento pai (que acaba separando o grupo de formulários) e o elemento filho (o elemento menor, que vai contér um pedaço do formulário).

O Container

O container é que vai determinar o flexbox e distribuir os elementos de formulário. Se form-grupo estiver ativo no mapa $uso-classes (nas configurações) serão criadas duas classes de container para serem usadas:

.form-container
Este container permite que coloque todo o formulário dentro dele. Ele tem margem negativa e padding para alinhamento. Se quiser usar fundo, borda ou sombra não é o método adequado (a menos que se acrescente um elemento pai com estas configurações). A grande vantagem deste container é que evita o excesso de divs dentro do HTML, deixando-o mais limpo e de fácil manutenção
.form-container--linha
Este container é para ser usado para cada linha de formulário. A diferença é que vai deixar o HTML maior (com mais divs). No entanto tem a grande vantagem de permitir que formulários sem tamanho determinado, ajustando-se ao espaço restante (flex: 1).

Caixas de formulário

Cada pedaço de formulário deve receber um agrupamento (através de uma div) que deve contér o label e o input. Por padrão, o Ruby já faz esta separação, então basta utilizar a que ele cria, modificando a classe conforme o tamanho.

Cada caixa vai receber uma classe .field-x onde x é o tamanho dela de 1 a 12. Cada valor é 1/12 do tamanho máximo (valor escolhido por ser divísivel por 3 e 4, permitindo a flexibilidade na formatação. Cada valor equivale a uma medida num sistema de grid).

Se você o elemento for filho direto de um .form-container--linha ele pode ser apenas .field. Neste caso o elemento vai esticar para caber no que sobrar do elemento pai.

Nesta caixa você também colocará a classe que formata o elemento filho (mais detalhes em AQUI, LINK TEMPORÁRIO).

Se quiser inserir um espaço em branco entre dois elementos, coloque a classe .pre-x. O x é a mesma medida de um field demonstrado acima. No entanto, para usar estas classes é necessário "ativar" no arquivo de configuração.

Formulários

Abaixo você verá os diversos elementos de formulários e como utilizá-los. Os tipos e algumas configurações especiais podem ser encontradas no arquivo de configuração

Para que os formulários sejam formatados direito siga os passos abaixo:

  1. Cada grupo de elemento (label + input) deve estar dentro de uma div. Não pode ter mais de um label e nem um elemento de formulário na div
  2. Essa div vai ter a classe que identifica e formata o elemento (ver mais detalhes abaixo)
  3. Se o código for gerado automaticamente pelo Ruby, retire o <br /> que ele acrescenta depois do label
  4. O uso de <p> dentro do div é desnecessário
  5. O div também pode ganhar a classe de medida field-x conforme exemplicado em Grupo de Formulários
  6. Algumas formatações especiais visuais são feitas colocando o label depois do input. Apesar de aumentar a experiência de uso, pode prejudicar quem usa leitores de tela. Use este recurso com cuidado.

Algumas funções são usadas em partes específicas do Front-Alego e outra é para uso dentro seu CSS.

Sprites

O primeiro ponto, evidentemente, é ter o arquivo de sprites gerados externamente, em um arquivo único. Este arquivo pode ser png, gif ou svg. No original crie uma grid e encaixe bem as imagens nela. O ideal é que cada imagem ocupe 100% da grid na largura ou altura. O ideal, no entanto, é que a imagem ocupe o espaço completo.

É possível acrescentar imagens menores ou maiores que a grid, mas use uma proporção lógica (75% do largura original, o dobro do tamanho e assim por diante). Use estes casos para economizar na quantidade de imagens do arquivo.

Depois de criado o arquivo, vá no arquivo de configurações e descomente sprite no mapa (ele vem comentado por padrão pois não é todo o sistema que faz uso de sprites).

No arquivo de configuração determine a extensão usada e o tamanho de cada sprite.

Criando as classes e determinando a posição

As classes são criadas no mapa $sprites no arquivo de configuração. Use nomes fáceis de serem lembrados e que identifique do que se trata aquele sprite (exemplo: um ícone de computador poderia ser nomeado como computador).

Para determinar a posição, utilize as coordenadas X, Y conforme a figura abaixo:

Para os exemplos abaixo, imagine que $sprite-medida: 150px.

Uma imagem que esteja na primeira coordenada (linha 1, coluna 1), seria:

SASS

$sprites: (

img-1: (1, 1),

);

O que resultaria no seguinte CSS:

CSS

.sprite-img-1 {

background: url(sprite.png) no-repeat 0px 0px;

width: 150px;

height: 150px;

}

Já uma imagem na primeira linha, porém segunda coluna ficaria:

SASS

$sprites: (

img-2: (2, 1),

);

O que resultaria no seguinte CSS:

CSS

.sprite-img-2 {

background: url(sprite.png) no-repeat -150px 0px;

width: 150px;

height: 150px;

}

Se precisar de um sprite com o dobro de tamanho:

CSS

}

O que resulta em:

CSS

.sprite-img-3 {

background: url(sprite.png) no-repeat 0px 0px;

width: 300px;

height: 300px;

}
